Тема: Retry-модель delivery_status=failed как временное состояние
Контекст: После первой интеграционной проверки failed использовался как временная ошибка, но это не было явно формализовано в application-логике.

Ограничения: Нет scheduler/backoff и нет persistence-очереди; retry инициируется orchestrator/use case.
Что учесть позже: Зафиксировать SLA-времена между попытками и правила reopen после manual_review.

Дата: 2026-04-13
Ветка: work
Тема: Decision-объекты policy обязаны явно нести маршрут next_channel
Контекст: В delivery logic обнаружено дублирование fallback-расчета: policy принимала решение, но use case повторно вычислял канал через FallbackPolicy.
Варианты:
1) Оставить неявный fallback через отдельные вызовы fallback-policy в use cases.
2) Включить результат маршрутизации в DeliveryDecision (поле next_channel) и исполнять его без повторного расчета.

Последствия: DeliveryPolicy закреплена как единая точка business-решений после неуспешной попытки; use cases исполняют decision и не содержат локального fallback-ветвления.
Ограничения: Поле next_channel описывает только следующий шаг в рамках текущего sync runtime; стратегия очередей/scheduler не вводилась.
Что учесть позже: При появлении конфигурируемой маршрутизации расширять decision-контракт, не возвращая вычисления каналов в use cases.

[Решение]
Дата: 2026-04-13
Ветка: work
Тема: In-memory runtime/persistence/queue как обязательный промежуточный этап
Контекст: После стабилизации domain+application+policy слоев нужен исполняемый контур, но подключение PostgreSQL/broker/scheduler на этом шаге преждевременно.
Варианты:
1) Сразу перейти к production-стеку (PostgreSQL + broker + workers).
2) Ввести синхронный in-memory runtime с явными repository/queue интерфейсами.
Почему выбран этот вариант: Позволяет запустить полный delivery flow, сохранить границы слоев и снизить сложность изменения бизнес-правил до подключения тяжелой инфраструктуры.
Последствия:
- Введены DeliveryRuntime, InMemoryDeliveryCardRepository, InMemoryDeliveryQueue.
- Execution loop зафиксирован как синхронный (seed -> run_once -> summary).
- Контракт репозитория вынесен в application/interfaces, а реализация остается в infrastructure.
Ограничения:
- Нет устойчивости к рестарту процесса.
- Нет distributed queue и межпроцессной координации.
- Нет real scheduler/backoff.
Что учесть позже: При переходе к real persistence сохранить сигнатуры runtime-контракта и не переносить policy-решения в инфраструктуру.

[ADR]
Дата: 2026-04-13
Тема: Сначала operator API только read-only, затем command-side
Контекст: После подключения persistence появился запрос на операторскую наблюдаемость, но write-команды преждевременны и рискованны до формирования регламента ручного вмешательства.
Решение:
- На текущем этапе введен только read-only operator API (список карточек, карточка по id, summary).
- Командные endpoint-ы (retry/manual status transitions/fallback overrides) отложены на отдельный блок.
- Query-логика вынесена в отдельный DeliveryCardReadService, а presentation не работает напрямую с ORM/repository-деталями.
Почему выбран этот вариант:
- Снижает риск нарушения бизнес-инвариантов доставки из-за преждевременных manual-команд.
- Позволяет быстро дать operator visibility по данным persistence.
- Сохраняет чистые границы слоев: presentation -> query service -> repository.
Последствия:
- Появился стабильный read контракт для будущей operator panel.
- Write-side ответственность остается в текущих use case/policy до проектирования command layer.
Ограничения:
- Нет ручных операторских действий через API.
- Нет аутентификации/авторизации enterprise-уровня.
Что учесть позже:
- Добавить command layer с явными safety-ограничениями, аудитом и idempotency для ручных действий.

[ADR]
Дата: 2026-04-14
Тема: Введение синхронного Scheduler/Process Manager после подключения persistence
Контекст: После появления PostgreSQL-персистентности runtime перестал быть строго одноразовым, но отсутствовал управляемый lifecycle для повторной обработки `waiting_retry` карточек.
Решение:
- Ввести отдельный инфраструктурный слой process manager (DeliveryProcessManager) поверх DeliveryRuntime.
- Вынести выбор кандидатов в отдельный selection/query компонент DeliveryRuntimeSelector.
- Оставить retry/fallback/deduplication решения в существующих policy/use-case модулях application-слоя.
Почему выбран этот вариант:
- Закрывает операционный разрыв между «есть сохраненное состояние» и «нет повторного исполнения».
- Сохраняет слоевые границы: scheduler не принимает бизнес-решения.
- Избегает преждевременной production-сложности (broker/workers/distributed orchestration).
Последствия:
- Runtime lifecycle поддерживает run_once, run_retry_cycle, run_all_pending.
- Появилась единая ProcessManagerSummary для локального контроля execution.
Ограничения:
- Нет time-based расписания и background исполнения.
- Нет distributed lock/coordination и broker-based очередей.
Что учесть позже:
- При переходе к real transport и command-side добавить безопасные operator-команды и наблюдаемость, не размывая policy-границы.

[ADR]
Дата: 2026-04-14
Тема: Real transport integrations после scheduler/persistence с сохранением stub режима
Контекст: После стабилизации domain/application/policies и появления persistence+process manager требуется подключить реальные каналы без срыва локального и тестового контура.
Решение:
- Подключить реальные адаптеры Renovatio/MAX/Email только после runtime/persistence слоя.
- Сохранить dual-mode `stub|real` и переключать его через env (`SLD_INTEGRATION_MODE`).
- Оставить MAX primary каналом, email — fallback, а policy-решения не переносить в transport.
Почему выбран этот вариант:
- Снижает риск архитектурной деградации: business-правила остаются в application/policy.
- Позволяет запускать CI и локальную разработку без внешних систем через stub режим.
- Даёт рабочий real transport контур для следующего этапа operator/patient функций.
Последствия:
- Контейнер собирает разные реализации адаптеров в зависимости от режима.
- Добавлен отдельный integration error mapping и env-конфиг внешних систем.
Ограничения:
- Это не production hardening: отсутствуют live-операционные гарантии, мониторинг и advanced security.
