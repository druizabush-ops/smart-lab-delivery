## ADR: Session-based Renovatio source for patient results
- Статус: Accepted (2026-04-21)
- Контекст: Старый `/patient/results` использовал internal delivery-card read model и `patient_id` как контекст.
- Решение:
  1) patient-facing results читаются только по server-side session;
  2) source of truth для patient API — Renovatio patient-facing методы по `patient_key`;
  3) internal delivery read model остается только для internal/operator контуров;
  4) backend contracts унифицированы как shared foundation для MAX bot UI и mini app UI.
- Последствия: снижается риск подмены контекста через query params; ускоряется разработка двух UI каналов поверх единого API.

[ADR]
Дата: 2026-04-21
Тема: PDF результата пациента извлекается из getPatientLabResultDetails -> data.files[0]
Контекст: На живом контуре подтверждено, что `getPatientLabResultDetails` возвращает base64 PDF в `data.files`, и для текущего сценария отдельный `file_id` не требуется.
Решение:

- base64 декодируется на server-side;
- наружу в patient API отдается бинарный `application/pdf`, а не base64-строка.
Почему выбран этот вариант:
- минимизирует доверие к клиенту и количество внешних вызовов;
- сохраняет единый session-based contour для bot UI и mini app.
Последствия:
- при множественных файлах действует правило «первый файл как основной PDF»;
- для альтернативного `file_id` flow потребуется отдельный ADR при изменении контракта Renovatio.

[ADR-022] Patient-facing result DTO normalization
Дата: 2026-04-22
Решение: list/details API возвращают patient-friendly DTO (`title`, `status`, `has_pdf`, `lab_name`, `clinic_name`) и не раскрывают сырой Renovatio payload.
Причина: UI mini app и будущий bot-first интерфейс должны быть стабильны и независимы от структуры Renovatio.

[ADR-023] Frontend never talks to Renovatio directly
Дата: 2026-04-22
Решение: любой результат/PDF route обслуживается только backend-слоем.
Причина: trust boundary, безопасность секретов и единая бизнес-логика.

[ADR-024] Backend owns patient session and external identity binding
Дата: 2026-04-22
Решение: session и привязка `external_platform_user_id -> session_id` живут только на сервере; frontend не хранит credentials.
Причина: foundation для no-repeat-login и безопасного lifecycle (logout/unbind/change credentials).

[ADR-025] Единая нормализация patient results contract для list/details/pdf
Дата: 2026-04-22
Решение:
- result_id нормализуется в application-слое единым правилом и для details обязателен fallback на `requested_result_id` из route.
- status в patient-facing DTO нормализуется единообразно по сырым статусам/ready-признакам и не вычисляется из UI.
- has_pdf вычисляется единообразно через `has_pdf|is_pdf_available|files|files_count|documents`.
- `pdf_open_url`/`pdf_download_url` строятся только из непустого `result_id` в формате `/patient/results/{id}/pdf`.
Причина: устранить расхождения list/details и предотвратить генерацию PDF route с пустым id.
Последствия:
- frontend перестает «догадываться» о статусе/доступности PDF;
- backend держит единый source of truth для patient-facing контракта.

[ADR-026] PDF open/download в mini app только через backend session endpoint
Дата: 2026-04-22
Решение:
- open/download сценарии используют единый fetch к `/patient/results/{id}/pdf` с `credentials: include`.
- Открытие выполняется через Blob URL после backend-ответа, скачивание — через тот же Blob без прямой работы с base64.
- Пользователю показываются только patient-friendly тексты ошибок без сырых JSON payload.
Причина: сохранить server-side session semantics в webview и исключить утечку интеграционных деталей на клиент.
Последствия:
- стабильнее поведение в мобильном webview;
- сохранена граница доверия (frontend не обращается к Renovatio напрямую).

[ADR-027] Каноническое сопоставление services + results для patient-facing показателей
Дата: 2026-04-22
Решение:
- человекочитаемая подпись показателя строится из `services` по шаблону `<parameter_code> • <человекочитаемое имя>`;
- сопоставление выполняется по коду (`parameter_name` / `parameter_code`), а не по индексу массивов;
- при отсутствии match используется fallback на исходный `parameter_name`.
Причина:
- сырой `parameter_name` часто непонятен пациенту, а индексное сопоставление нестабильно и приводит к регрессиям.
Последствия:
- в details UI закреплен формат `Название: значение единица`;
- mapping логика вынесена в отдельный frontend util-слой, не размазана по JSX.

[ADR-028] Честная status-нормализация для patient-facing list/details
Дата: 2026-04-22
Решение:
- если payload уже содержит реальные показатели (`results`/`indicators`) и/или доступный PDF, status нормализуется в `Готов`;
- единая логика действует одинаково в list и details DTO.
Причина:
- исключить misleading-состояние `В обработке` для фактически готовых результатов.
Последствия:
- пациент видит консистентный и фактический статус;
- снижена вероятность UI-расхождений между list/details.
