Название: Централизованная модель статусов Smart Lab Delivery
Дата актуализации: 2026-04-13

Цель документа
Зафиксировать единый словарь статусов и разрешенные переходы, чтобы избежать хаотичных строковых значений и расхождений в логике.

1. Группы статусов

1.1 Статус лабораторного результата (lab_result_status)
- pending: результат еще не готов в МИС.
- ready: результат готов и доступен для доставки.
- missing_pdf: результат есть, но PDF отсутствует или недоступен.
- blocked: результат временно заблокирован к доставке (например, валидация данных не пройдена).

1.2 Статус карточки очереди (queue_status)
- active: карточка в активной обработке.
- waiting_retry: ожидает повторной попытки.
- manual_review: передана на ручную обработку оператору.
- done: доставка завершена (успешно или финально закрыта по политике).

1.3 Статус доставки (delivery_status)
- not_started: доставка еще не началась.
- max_sent: успешно отправлено через MAX.
- email_sent: успешно отправлено через email.
- failed: последняя попытка завершилась ошибкой, статус трактуется как временная неудача.
- exhausted: попытки исчерпаны, автоматическая доставка остановлена.

1.4 Результат попытки отправки (attempt_status)
- success: попытка успешна.
- error: попытка завершилась ошибкой.

2. Назначение статусов
- lab_result_status отражает готовность медицинского результата к процессу доставки.
- queue_status отражает операционный этап карточки в очереди обработки.
- delivery_status отражает итог канала/попыток доставки для карточки.
- attempt_status отражает терминальный исход отдельной зафиксированной попытки отправки.

3. Разрешенные переходы
- lab_result_status: pending -> ready | blocked; ready -> missing_pdf | blocked.
- queue_status: active -> waiting_retry | manual_review | done; waiting_retry -> active | manual_review | done; manual_review -> active | done.
- delivery_status: not_started -> max_sent | email_sent | failed; failed -> not_started | exhausted.
- attempt_status: переходов нет (фиксируется только терминальный исход новой попытки).

4. Retry / fallback трактовка
- failed + waiting_retry означает временную ошибку, после которой допустима новая попытка.
- Повторная попытка возвращает delivery_status в not_started и queue_status в active до следующего send.
- Если лимиты попыток исчерпаны, карточка переводится в exhausted и queue_status=done.
- Основной канал: MAX; fallback-канал: EMAIL.
- Fallback MAX -> EMAIL выполняется только однократно и только после неуспешной работы MAX.
- Если EMAIL тоже завершился ошибкой, карточка переходит в manual_review (без обратного переключения в MAX).

5. Правила deduplication
- Повторная отправка запрещена для max_sent/email_sent и exhausted.
- Наличие хотя бы одной успешной попытки в attempts блокирует новые send-операции.
- Повторная попытка после error допустима только при не-терминальном delivery_status и в рамках RetryPolicy.

6. Запрещенные переходы (минимальный набор)
- queue_status: done -> active (без явного ручного reopen-сценария).
- delivery_status: max_sent/email_sent -> failed (исторически фиксируется новой попыткой, а не откатом статуса).
- attempt_status: success -> error внутри одной попытки запрещен.
- attempt_status: error -> success внутри одной попытки запрещен (повтор — это новая попытка).

7. Правила изменения статусов
- Изменение статусов допускается только через доменные/прикладные механизмы, где сохраняется причина перехода.
- Каждый переход должен быть прослеживаемым в журнале событий доставки.
- Любое изменение словаря статусов или переходов требует одновременного обновления:
  1) доменной реализации статусов;
  2) этого документа;
  3) docs/PROJECT_JOURNAL.txt;
  4) docs/DECISIONS.txt (если изменение архитектурно значимо).

8. Централизация и запрет на распыление
- Статусы нельзя разбрасывать строками по коду.
- Использование «сырых» строк статусов в логике запрещено.
- Единый источник истинных значений статусов — доменный слой (domain/statuses) и этот документ как его нормативное описание.

9. Ограничения текущего этапа
- Retry/fallback реализованы как синхронные policy-решения без scheduler, очередей и real-time исполнения.
- Deduplication действует в рамках текущего runtime-состояния карточки без persistent-хранилища.
