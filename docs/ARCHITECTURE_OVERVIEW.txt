Название: Архитектурный обзор Smart Lab Delivery
Дата актуализации: 2026-04-13

1. Архитектурная цель
Разделить систему на стабильные слои, чтобы бизнес-логика доставки результатов не зависела от конкретного UI, транспорта и внешних API-деталей.

2. Логическая схема слоев
- integration: получение и нормализация данных из МИС Renovatio.
- domain: сущности, контракты, статусы, инварианты.
- application: оркестрация сценариев формирования очереди и доставки.
- infrastructure: технические реализации каналов (MAX/email), хранение, логирование.
- presentation: операторская панель и интерфейс управления исключениями.

3. Основные потоки
1) integration/renovatio получает данные пациента и результатов.
2) application/orchestration создает/обновляет карточку доставки.
3) application/delivery выбирает канал (MAX -> email fallback).
4) infrastructure фиксирует попытку, результат и ошибки.
5) presentation/operator_panel показывает актуальное состояние оператору.

4. Архитектурные инварианты
- Источник медицинских данных — только Renovatio.
- Статусы централизованы в domain/statuses и не дублируются строками по коду.
- Решения по ретраям и ручной обработке принимаются в application, а не в инфраструктурных адаптерах.
- Логи попыток отправки неизменяемы (append-only подход на уровне модели событий).
- MAX является основным каналом доставки, email используется только как резервный канал.

5. Границы текущего этапа
На текущем этапе зафиксирован каркас и документация. Реализация интеграционных клиентов, очереди, БД, UI и отправки будет добавляться итеративно после согласования контрактов.

6. Риски, учтенные при проектировании
- Риск расползания бизнес-логики в адаптеры каналов.
- Риск несогласованных статусов между модулями.
- Риск потери контекста без непрерывного журнала.
- Риск изменений архитектуры из-за отсутствия финальных контрактов Renovatio.

7. Открытые вопросы архитектуры
- Нужен ли outbox-паттерн в первом релизе или он переносится на следующий этап?
- Где проходит граница между audit log оператора и журналом технических событий доставки?
- Какой формат версионирования контрактов интеграции будет принят?

9. Runtime execution contour (текущий этап)
- runtime/infrastructure:
  - DeliveryRuntime — синхронный execution service уровня process_cycle.
  - InMemoryDeliveryQueue — FIFO очередь id карточек.
  - InMemoryDeliveryCardRepository — операционное in-memory хранилище карточек.
  - build_operational_card_id (infrastructure/identity) — единая точка вычисления operational card id для runtime/repository.
- execution pipeline:
  1) seed_ready_results() получает READY результаты через orchestrator;
  2) создаёт/сохраняет карточки и ставит их id в очередь;
  3) run_once() последовательно обрабатывает очередь;
  4) обновленные карточки сохраняются в repository;
  5) DeliveryRunSummary возвращается как operator-facing снимок цикла.

10. Обновленная зависимость слоев
- integration -> (данные/каналы) -> application use cases
- application -> (правила/policy/orchestration) -> domain
- infrastructure/runtime -> использует application + integration + domain контракты
- domain/application не зависят от infrastructure

11. Границы runtime-блока
- Не подключаются PostgreSQL/Redis/Rabbit/Kafka.
- Не вводятся async workers, cron, multiprocessing.
- Репозиторий и очередь не принимают бизнес-решения retry/fallback/dedup.


12. Persistence / database contour (новый блок)
- Выбран стек SQLAlchemy + Alembic + PostgreSQL.
- Domain и ORM разделены: domain-модели живут в src/domain, persistence-модели — в src/infrastructure/persistence/models.
- Явный mapping domain <-> persistence реализован в src/infrastructure/persistence/mappers.
- Реальный репозиторий PostgresDeliveryCardRepository подключается как альтернатива InMemoryDeliveryCardRepository по флагу SLD_REPOSITORY_MODE.
- Runtime поддерживает dual-mode: in_memory (по умолчанию) и postgres (устойчивое состояние между запусками).

13. Operator API (read-only) слой
- Добавлен presentation-слой src/presentation/operator_api на FastAPI.
- Текущий API-контур ограничен query-доступом и не содержит write endpoint-ов.
- Схема зависимостей:
  1) presentation/operator_api/routers ->
  2) application/services/DeliveryCardReadService ->
  3) infrastructure repository (in-memory или postgres).
- API не использует ORM-модели напрямую: наружу возвращаются отдельные response schema (Pydantic).

14. Read-only контракты operator API
- GET /operator/cards
  - Возвращает список карточек.
  - Поддерживает фильтры: status, queue_status, channel, patient_id, lab_result_id, only_active, only_done.
- GET /operator/cards/{card_id}
  - Возвращает карточку по deterministic operational id.
- GET /operator/cards/summary
  - Возвращает агрегированную сводку состояния карточек.

15. Границы operator API этапа
- Не добавляются retry/fallback/manual command endpoint-ы.
- Не добавляются edit/delete операции.
- Не добавляются enterprise auth/roles/permissions.
- API создан как foundation для будущей operator panel и command-side блока.

16. Scheduler / Process Manager слой (новый блок)
- Добавлен синхронный lifecycle-слой поверх runtime + persistence:
  - DeliveryRuntimeSelector (query/select логика кандидатов);
  - DeliveryProcessManager (управление execution cycle);
  - ProcessManagerSummary (operator/debug сводка выполнения).
- Execution lifecycle теперь поддерживает три явных режима:
  1) run_once(patients): только discovery READY результатов и один цикл обработки;
  2) run_retry_cycle(): только повторная обработка retry-кандидатов из persistence;
  3) run_all_pending(patients): discovery новых карточек + обработка всех pending-кандидатов.
- Граница ответственности:
  - selector выбирает карточки по status/queue_status и не принимает business-решений;
  - process manager управляет последовательностью циклов и агрегирует summary;
  - retry/fallback/deduplication остаются в policy/use-case слое (application).
- Persistence выступает источником операционного состояния между циклами, а не scheduler-policy движком.
- Ограничения этапа:
  - нет async/background workers, cron и broker-оркестрации;
  - нет distributed scheduling и time-based backoff;
  - execution остается локальным, синхронным и воспроизводимым.
