Название: План тестирования Smart Lab Delivery
Дата актуализации: 2026-04-13

1. Уровни тестирования
- Unit / domain: инварианты сущностей, валидаторы, статусные переходы.
- Unit / application: policy-решения retry/fallback/deduplication и use-case orchestration.
- Integration: проверка stub-адаптеров и сквозного контура AppContainer + DeliveryOrchestrator.

2. Критичные сценарии
- Валидность Patient/LabResult/DeliveryAttempt/DeliveryCard и запрет невалидных состояний.
- Согласованность DeliveryCard status/queue_status при add_attempt и change_status.
- Retry limits (total/MAX/EMAIL), terminal-status блокировки и manual_review ограничения.
- Fallback только MAX -> EMAIL; после failure в EMAIL — только manual_review.
- Deduplication: запрет на повторную отправку при success/exhausted/terminal.
- Use-case контур: policy решает, use case исполняет; без локального пересчета fallback.
- Интеграционный поток: ready-results из Renovatio -> create/process/retry в orchestrator.

3. Границы unit/integration
- Unit:
  - Тестируются pure business-правила и orchestration решения без внешних систем.
  - Зависимости в use cases подменяются test-stub объектами.
- Integration:
  - Используются существующие in-memory deterministic stub-адаптеры.
  - Реальный HTTP/БД/очереди/runtime scheduler не используются.

4. Что пока не тестируется
- Реальные сетевые интеграции Renovatio/MAX/SMTP.
- Персистентность и миграции БД (частично покрыто unit-тестами mapper/repository).
- Конкурентная обработка/очереди/асинхронный runtime.
- UI и операторская панель.
- Нагрузочные и chaos-тесты.

5. Регрессионно-опасные зоны
- Изменение правил delivery_status/queue_status в DeliveryCard.
- Изменение приоритетов/лимитов retry/fallback, приводящее к ping-pong маршрутизации.
- Дублирование policy-логики в use cases.
- Изменение deterministic-поведения stub-провайдеров без обновления тестов.

6. Минимальный регрессионный набор (обязательный запуск)
- `pytest tests/unit/domain`
- `pytest tests/unit/application`
- `pytest tests/integration`
- `pytest`


7. Persistence тесты (новый минимум)
- Unit: `tests/unit/infrastructure/persistence/test_delivery_card_mapper.py`
  - корректный mapping DeliveryCard/DeliveryAttempt в ORM-модели и обратно.
- Repository: `tests/unit/infrastructure/persistence/test_postgres_delivery_card_repository.py`
  - save/get/update/list_by_queue_status;
  - сохранение и чтение attempts;
  - консистентность deterministic card id.
- Ограничение текущего контура: repository тестируется через SQLite in-memory surrogate; отдельный PostgreSQL integration-job выделен в tech debt.

8. Operator API read-only тестовый контур
- Integration: `tests/integration/test_operator_api_read_only.py`
  - `GET /operator/cards` возвращает список карточек;
  - `GET /operator/cards/{card_id}` возвращает карточку по id;
  - `GET /operator/cards/summary` возвращает агрегаты;
  - фильтрация по `status` работает корректно;
  - пустое хранилище обрабатывается без падений;
  - API работает с PostgresDeliveryCardRepository (sqlite surrogate в тесте).

9. Критичность для operator read access
- API должен оставаться read-only (без write endpoint-ов).
- Query-слой должен быть отделен от роутеров (без прямого ORM-ответа наружу).
- Контур должен работать в обоих режимах repository: in-memory и postgres.

10. Покрытие scheduler/process manager (новый блок)
- Unit / infrastructure: `tests/unit/infrastructure/test_delivery_process_manager.py`
  - process manager корректно подбирает `waiting_retry` карточки через selector;
  - `done/manual_review/exhausted` исключаются из повторной обработки;
  - `run_once` не ломает текущий single-cycle сценарий;
  - summary (`ProcessManagerSummary`) отражает фактически обработанные карточки;
  - selector совместим с postgres repository режимом (sqlite surrogate в тесте).

11. Регрессионно-опасные сценарии повторной обработки
- Ошибочное попадание terminal/manual карточек в retry-cycle.
- Дублирование policy-решений внутри scheduler/process manager.
- Неконсистентные summary-метрики при смешанном цикле (новые + существующие карточки).
- Нарушение dual-mode repository поддержки (in-memory / postgres).

12. Покрытие real integration слоя (без live-вызовов)
- Unit / integration: `tests/unit/integration/test_renovatio_real_adapter.py`
  - request/response mapping Renovatio;
  - обработка `error` поля и bad-response сценариев.
- Unit / integration: `tests/unit/integration/test_max_real_adapter.py`
  - сборка запроса к `/messages`;
  - заголовок `Authorization`;
  - обработка отсутствия patient->recipient mapping.
- Unit / integration: `tests/unit/integration/test_email_real_adapter.py`
  - построение SMTP-сообщения;
  - subject/body/sender поля;
  - проверка успешного и error-потока через mock SMTP.
- Unit / integration: `tests/unit/integration/test_container_dual_mode.py`
  - wiring контейнера для `SLD_INTEGRATION_MODE=stub|real`.

13. Что сознательно не тестируется live
- Реальные интернет-вызовы в Renovatio/MAX/SMTP.
- Реальные rate-limit сценарии MAX (30 rps) в CI.
- Полный e2e mini app UI-поток.

14. Покрытие operator command layer
- Unit: `tests/unit/application/test_operator_commands.py`
  - policy/guard допустимости OperatorActionPolicy;
  - retry в допустимом состоянии;
  - retry в недопустимом состоянии (manual_review);
  - manual_review command;
  - requeue command;
  - override channel command;
  - запрет операторских действий для terminal статусов.
- Integration: `tests/integration/test_operator_api_read_only.py`
  - command endpoint-ы возвращают корректные HTTP-коды;
  - допустимые команды меняют состояние карточки;
  - недопустимая команда возвращает 409, без 500;
  - read-only endpoint-ы остаются согласованными после command-вызовов.

15. Регрессионно-опасные сценарии operator commands
- Случайный обход OperatorActionPolicy через роутер или инфраструктуру.
- Неконтролируемый requeue карточек из terminal/done состояний.
- Override канала после terminal success/exhausted.
- Расхождение read модели и фактического состояния карточки после command-вызова.

16. Покрытие patient mini app / MAX UI layer
- Frontend unit:
  - рендер списка результатов;
  - рендер empty/no-results;
  - рендер backend error state;
  - чтение `start_param` и переход к details;
  - fallback поведение вне MAX (`window.WebApp` отсутствует).
- Frontend API client:
  - успешный ответ;
  - ошибка backend;
  - пустой результат.
- Backend integration:
  - `GET /patient/results` фильтрует данные по `patient_id`;
  - `GET /patient/results/{id}` возвращает только patient-owned карточку;
  - patient response-контракт не равен operator response-контракту (без полного attempts history).

17. Что пока не автоматизировано в mini app блоке
- E2E тесты запуска внутри реального MAX клиента.
- Скриншотные/визуальные regression тесты с реальным `@maxhub/max-ui` SDK.
- Криптографическая валидация `initData` (покрывается будущим backend security блоком).
