Название проекта: Smart Lab Delivery

1. Цель проекта
Создать устойчивую систему автоматической доставки лабораторных результатов пациентам с минимальным ручным участием и прозрачной операторской обработкой исключений.

2. Бизнес-суть проекта
- Источник данных: МИС Renovatio.
- Основной канал отправки: MAX.
- Резервный канал отправки: email.
- Система должна формировать внутреннюю очередь карточек доставки, хранить статусы, историю попыток и действий оператора.

3. Основные правила проекта
- Явность важнее краткости.
- Все значимые изменения фиксируются в журнале.
- Комментарии в коде и документация ведутся на русском языке.
- Статусы централизуются в доменном слое.
- Временные решения обязательно маркируются как технический долг.

4. Архитектурные принципы
- МИС Renovatio — источник истины.
- Внутренний сервис — оркестратор процесса доставки.
- Разделение на слои: integration / application / domain / infrastructure / presentation.
- Бизнес-логика не смешивается с транспортом и UI.

5. Границы проекта
В рамках текущего этапа создан архитектурный каркас, документационная база и правила сопровождения. Сложная бизнес-логика сознательно отложена до согласования контрактов и сценариев.

6. Основные сущности системы
- Пациент
- Лабораторный результат
- Карточка доставки
- Попытка отправки
- Ошибка доставки
- Операторский комментарий

7. Статусы и правила изменения статусов
Актуальная модель статусов и переходов ведется в отдельном документе: docs/STATUS_RULES.txt.

8. Основные интеграции
- МИС Renovatio (получение данных и документов)
- MAX (основная доставка)
- Email SMTP/API (резервная доставка)

9. Источники истины и потоки данных
- Данные пациента и результатов — из Renovatio.
- Внутренний реестр карточек — источник операционного состояния доставки.
- История отправок и ошибок — внутренний неизменяемый журнал событий.

Причина: Выявлены архитектурные расхождения: fallback-логика вычислялась в двух местах (DeliveryPolicy и HandleDeliveryFailureUseCase), а решение о следующем канале оставалось неявным.
Какие расхождения были найдены:
- Дублирование fallback-решений между policy-слоем и use case обработки failure.
- DeliveryDecision не нес явного результата о смене канала, из-за чего use cases могли повторно вычислять маршрут.
- Retry use case содержал собственную ветку retry-policy, что размывало границу «policy решает — use case исполняет».
