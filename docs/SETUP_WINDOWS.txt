Название: Настройка и проверка проекта в Windows 10 (PowerShell)
Дата актуализации: 2026-04-13

1. Требования
- Windows 10
- PowerShell 5.1+ или PowerShell 7+
- Git

7. Текущий запуск приложения
На текущем этапе полноценный runtime отсутствует: проект находится в фазе инженерной и архитектурной инициализации.

8. Что проверять на этом этапе
- Наличие обязательных документов в docs/.
- Соответствие структуры каталогов архитектурным слоям.
- Наличие записи в PROJECT_JOURNAL о произведенных изменениях.
- Отсутствие преждевременной реализации интеграционных клиентов, БД, UI и доставки.

9. Установка и запуск тестов (pytest)
```powershell
python -m pip install --upgrade pip
python -m pip install pytest
```

10. Запуск тестов в PowerShell
```powershell
# Все тесты
python -m pytest

# Только unit-тесты domain
python -m pytest .\tests\unit\domain

# Только unit-тесты application
python -m pytest .\tests\unit\application

# Только integration-тесты
python -m pytest .\tests\integration
```

11. Как читать результат
- `passed` — сценарий отработал по ожидаемому контракту.
- `failed` — зафиксирован регресс или нарушение инварианта.
- `xfailed/skipped` сейчас не используются в базовом контуре.
- Для детального вывода используйте:
```powershell
python -m pytest -vv
```

12. Запуск runtime cycle (PowerShell)
```powershell
# Один синхронный цикл runtime (seed + processing)
python .\scripts\run_runtime_cycle.py
```
Ожидаемый результат: вывод блока `Delivery Runtime Summary` с числом созданных/обработанных карточек и финальными счетчиками.

13. Проверка тестов после runtime/infrastructure изменений
```powershell
# Полный контур
python -m pytest

# Только runtime/infrastructure блок
python -m pytest .\tests\unit\infrastructure
```

14. Важные ограничения текущего runtime
- Хранилище и очередь in-memory: состояние не сохраняется после завершения процесса.
- Execution loop синхронный: фоновый scheduler/worker не реализован.
- Integration-клиенты остаются stub-адаптерами.

17. Env-переменные для postgres режима
```powershell
$env:SLD_REPOSITORY_MODE = "postgres"
$env:SLD_DB_HOST = "localhost"
$env:SLD_DB_PORT = "5432"
$env:SLD_DB_NAME = "smart_lab_delivery"
$env:SLD_DB_USER = "postgres"
$env:SLD_DB_PASSWORD = "postgres"
$env:SLD_DATABASE_URL = "postgresql+psycopg://postgres:postgres@localhost:5432/smart_lab_delivery"
```

21. Запуск operator API (read-only) в PowerShell
```powershell
# Установить зависимости (включая FastAPI/uvicorn)
python -m pip install -r .\requirements.txt

# Запуск API
python .\scripts\run_operator_api.py
```
Ожидаемо: Uvicorn поднимает сервер на `http://127.0.0.1:8001`.

22. Быстрая проверка endpoint-ов operator API
```powershell
# Список карточек
Invoke-RestMethod -Uri "http://127.0.0.1:8001/operator/cards" -Method GET

# Сводка
Invoke-RestMethod -Uri "http://127.0.0.1:8001/operator/cards/summary" -Method GET

# Карточка по id (пример)
Invoke-RestMethod -Uri "http://127.0.0.1:8001/operator/cards/<card_id>" -Method GET
```

23. Фильтрация в операторском API
```powershell
Invoke-RestMethod -Uri "http://127.0.0.1:8001/operator/cards?status=failed" -Method GET
Invoke-RestMethod -Uri "http://127.0.0.1:8001/operator/cards?queue_status=manual_review" -Method GET
Invoke-RestMethod -Uri "http://127.0.0.1:8001/operator/cards?only_active=true" -Method GET
```

24. API-тесты operator read-only слоя
```powershell
python -m pytest .\tests\integration\test_operator_api_read_only.py
python -m pytest
```

25. Ограничение текущего этапа operator API
- Endpoint-ы доступны только в read-only режиме.
- Ручные retry/fallback/status-change команды отсутствуют по дизайну текущего блока.
